# 0004. サイトの英語対応（i18n・ja ルート + /en/）

- 状態: Accepted
- 日付: 2026-07-06

## 背景

両アプリ（こうら日記・Schemely）は UI が日本語/英語対応。一方サポートサイトは日本語単独で、
App Store の審査担当・海外ユーザーが規約・プライバシー・紹介を読めなかった。

## 決定

- Astro ネイティブ i18n を使う。`defaultLocale:'ja'` / `locales:['ja','en']` /
  `routing.prefixDefaultLocale:false`。**日本語はルート維持**（`/schemely/terms`）、
  **英語は `/en/` 配下**（`/en/schemely/terms`）。
- UI チコル（nav・doc の定型文）は `src/i18n/ui.ts` の辞書で共通化し、共有レイアウト
  （BaseLayout / DocLayout）が locale で切り替える。**ランディング・規約の本文は locale 別の
  ページファイルに持つ**（長文は辞書に入れない）。
- ヘッダーに言語スイッチャ、各ページに `<html lang>` と `hreflang`（ja / en / x-default=ja）。
- 規約類の英訳には「**In case of any discrepancy, the Japanese version prevails**」を明記し、
  日本語版へリンクする。

## 理由

- 動機は集客（SEO）より **App Store 審査・海外ユーザーのサポート導線**。読めることが目的。
- `prefixDefaultLocale:false` なら既存の日本語 URL（App Store のポリシー URL 等）が変わらない。
- 機械翻訳の法務文書は法的リスクがあるため、「日本語版優先」を明記して権威を日本語版に置く。
- 稼働中の日本語問い合わせフォーム（Web3Forms 通知）を壊さないため、**フォームは共通
  コンポーネント化せず en ページを新規作成**した（ja は無変更）。en は件名に `(EN)` を付け、
  redirect を `/en/<app>/thanks` に向ける。

## 代替案

- **両 locale に prefix（/ja/ /en/）**: 対称だが既存 URL が変わり App Store 側の差し替えが要る。却下。
- **Accept-Language 自動リダイレクト**: 静的配信では Workers 実装が要り複雑・SEO 上も非推奨。却下。
- **フォームの共通コンポーネント化**: 重複は減るが稼働中 ja フォームに手を入れる risk。今回は見送り。

## 影響

- 新ページを足すときは ja（ルート）と en（`/en/`）の両方を用意する。未対応の locale があると
  言語スイッチャがリンク切れになる（現状は全ページ両 locale を用意済み）。
- ランディングは locale 別ファイルで本文が二重管理になる（小規模サイトゆえ許容・差分に注意）。
- 法務 3 種の英訳は「参考訳・日本語版優先」。内容変更時は日本語版を正として英語版を追随させる。
